# 07 — Internationalization (i18n)

This architecture uses a document-level internationalization pattern to support multiple languages.

## 1. Routing (Astro)

Astro handles the routing. We keep the default language (e.g., `en`) at the root `/` and prefix secondary languages (e.g., `/es`).

- **`astro.config.mjs`**: Set up the `i18n` property with your locales.
- **Helpers**: Create a helper like `getUrlPrefix(lang)` inside `src/i18n/index.ts` so all internal links automatically get the correct prefix based on the current page's language.

## 2. Sanity Schema

We use the `@sanity/document-internationalization` plugin in the Studio.

Instead of field-level translations (where every individual text field has an `en` and `es` version), we use **document-level translations**. This means we have a complete separate document for English (e.g., `homePage-en`) and a complete document for Spanish (`homePage-es`).

To configure this in the Studio:
1. Install the `@sanity/document-internationalization` plugin in the `/studio` workspace.
2. Configure it in `sanity.config.ts` to add the language selector to your singletons and collections.

## 3. Data Layer (`content.ts`)

Since we have one document per language, our fetchers need to know which language to query.

Update the fetchers in `src/lib/content.ts` to accept a `lang` parameter:

```typescript
export async function getHomeView(lang: string = 'en') {
  // Query the localized document ID
  const docId = `homePage-${lang}`;
  // Fetch from Sanity...
}
```

## 4. Caching & ISR (Crucial)

Singletons are request-cached via module-level promises to avoid hitting Sanity multiple times per build or request. 

**Single-language approach:**
```typescript
let homePromise: Promise<HomeView> | null = null;
```

**Bilingual approach:**
When supporting multiple languages, a single promise is not enough. You must use a `Map` to cache by language:
```typescript
let homePromises = new Map<string, Promise<HomeView>>();

export async function getHomeView(lang: string = 'en') {
  if (!homePromises.has(lang)) {
    homePromises.set(lang, fetchFromSanity(lang));
  }
  return homePromises.get(lang)!;
}
```

### Preventing Memory Leaks under ISR

If your site runs on Vercel as ISR (Serverless Functions regenerating in the background), the Vercel function can stay "warm" across multiple requests. This means the `homePromises` Map will **never clear itself**, and the server will serve stale content even after a Sanity webhook triggers a revalidation.

To fix this, you must clear the cache at the start of **every** request using Astro middleware:

1. In `src/lib/content.ts`, export a reset function:
```typescript
export function resetContentCache() {
  homePromises.clear();
  settingsPromises.clear(); // Add any other cached singletons here
}
```

2. Create `src/middleware.ts` to call this function:
```typescript
import { defineMiddleware } from 'astro:middleware';
import { resetContentCache } from './lib/content';

export const onRequest = defineMiddleware((context, next) => {
  // Clear the in-memory cache so fresh reads go to Sanity
  // (Sanity's own client caching / CDN is still respected if configured)
  resetContentCache();
  return next();
});
```

This setup guarantees that when the Vercel edge cache expires, the underlying function actually performs a fresh read from Sanity instead of returning its own warm memory.
